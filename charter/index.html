Goals

The Semantic Agent Communication Community Group seeks to define semantic models and communication structures that enable autonomous software agents to interoperate in a verifiable, accountable, and machine-interpretable manner. Whereas existing approaches focus primarily on transport protocols or model-level behaviors, this group concentrates on the semantic layer required for identity, delegation, capability discovery, intent modeling, and execution traceability.

The long-term objective is to provide a common semantic foundation that supports interoperable agent-to-agent communication across heterogeneous system architectures, identity frameworks, and governance requirements.

Scope of Work

The Community Group will define and maintain the formal ontology and semantic models necessary for AI Agent interoperability, security, and communication. The scope includes the following:

  • Interaction Semantics: Formal definitions for Intent, Delegation, Capability advertisement, ExecutionRecord, and related CommunicativeActs.
  • Identity and Verifiability Primitives: Use of Decentralized Identifiers (DIDs), Verifiable Credentials (VCs), signatures, attestations, and provenance structures needed to establish trust in agent actions. This includes the binding of DIDs to hardware roots of trust (e.g., ARM TrustZone) through attestation.
  • Narrative Model: Definition of an append-only, context-complete semantic log that provides traceability, accountability, and explainability of agent behavior.
  • Semantic Ledger Model: A logical model for representing immutable, ordered, verifiable records. The CG defines the conceptual structure without mandating implementation technologies (e.g., no blockchain requirement).
  • Conformance Definitions: Minimal normative requirements for interpreting CommunicativeActs, Delegations, and other semantic components consistently across implementations.
  • Machine-Readable Ontology Artifacts: RDF/OWL vocabularies, JSON-LD contexts, and SHACL shapes capturing the concepts described above in a formally verifiable manner.
  • AgentIDL (WebIDL Extension): An interface description language generated from the core ontology to support strongly-typed, semantically-aware code generation. The CG will define AgentIDL and produce a reference compiler implementation.
  • Knowledge Representation Technology Bridge: Specification of the AgentIDL interfaces and serialization rules needed to integrate stateful, version-controlled knowledge representation systems (e.g., graph databases, version-controlled knowledge stores). Stateless embedding/vector stores are out of scope.
  • Schema.org Extension Submission: Drafting proposed extensions to Schema.org incorporating agent semantics for broader discoverability and interoperability.

Out of Scope

The following are explicitly out of scope for this Community Group:

  • Transport Protocols (HTTP, WebSockets, TCP, UDP, QUIC)
  • Internal Agent Reasoning Models (LLMs, inference engines, planners, cognitive architectures)
  • Execution Backends / Compute Targets (CPU, GPU, WASM, runtime specifics)
  • Economic or Settlement Systems
  • Blockchain / Distributed Ledger Protocols
  • Domain-Specific Knowledge Models
  • Legal or Policy Interpretation
  • Executable Runtimes, SDKs, or Complete Agent Systems
  • Conflict Resolution Algorithms (CRDT, OT, etc.)
  • Browser Applications and Short-Lived Client Interactions

Deliverables

Specifications

  • AI Agent Semantic Communication Ontology v1.0 (Turtle and JSON-LD serialization)
  • AgentIDL Specification and Reference Compiler
  • SHACL Shapes for Ontology Validation
  • Implementation and Governance Notes
  • Interoperability Best Practices for Agent Runtimes

Non-Normative Community Group Reports

  • Use cases and requirements
  • White papers and guidance documents
  • Interoperability analysis and mappings

Test Suites and Example Implementations

The group MAY produce test suites or non-normative example implementations to support specifications. These outputs will be governed by the repository LICENSE file.

Roles and Participation

Chairs

The Community Group will be led by Chair(s) selected by group participants. If participants from five distinct organizations call for an election, the CG will conduct a formal election in consultation with the W3C Community Development Lead.

Editors

Editors are responsible for maintaining the active specifications and ontology deliverables.

Contributors

Contributors may submit issues, comments, and pull requests. Substantive contributions require agreement to the W3C Community Contributor License Agreement (CLA).

Implementers

Organizations or individuals developing implementations or testbeds based on the group’s deliverables may participate as Implementers.

Community Participants

All participants must follow the W3C Code of Ethics and Professional Conduct.

Dependencies or Liaisons

  • Credentials Community Group
  • Decentralized Identifier Working Group
  • Verifiable Credentials Working Group
  • Web of Things Working Group
  • AI governance and safety-related community efforts
  • External agent framework implementer communities

Community and Business Group Process

The Group operates in accordance with the W3C Community and Business Group Process. Any Charter content inconsistent with that Process is void.

Specifications and software outputs MUST use the W3C Software and Document License.

All repositories MUST include:

  • CONTRIBUTING
  • LICENSE

Transparency

All technical work will be conducted in public GitHub repositories. Meeting minutes and summaries will be posted publicly.

Decision Process

The group will seek consensus. When consensus is unclear, the Chair(s) MAY issue a Call for Consensus (CfC). If consensus still cannot be reached, Committers MAY select a direction with substantial support. Objections MAY be addressed through alternative forks or separate notes.

Amendments to this Charter

Amendments require a 30-day vote and support from at least two-thirds of votes cast. Minor corrections MAY be performed through standard group decisions.
